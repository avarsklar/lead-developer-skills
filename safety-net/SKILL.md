# /safety-net

You are setting up the **safety net** under an app a non-technical person built with AI and put live. This is a one-time hardening pass. When it's done, they **can't lose their work and can't lose their data** — every future change is recoverable.

## First — is this a fresh setup or a return trip? Check for unfinished work before you state the promise

This skill is written as a one-time pass, but in practice builders come back to it: a step got deferred, a 🔴 item was left "for later," a follow-up branch was drafted and never pushed. The floor may already be *half* there. So before you frame the promise as if from zero, **look for what's already in place and what was left open** — read `stack.md`, `safety-net.md`, and especially any **`## Still open`** ledger inside `safety-net.md` (the durable, dated list of deferred items the completion rule makes you write on the way out).

- **Fresh project (none of those files exist):** run the whole skill as written below.
- **Return trip with open items:** do **not** re-walk finished steps or re-state the from-scratch promise — that buries the very thing they came back for. Open with the **unfinished list first, loudest item on top**: *"Last time we set up most of your safety net but left these open — let's close them now."* Render those open items as *this run's* checklist. A deferred 🔴 (an unrotated signing key, a guessable secret still live) leads — it is the reason you're back.

The failure this prevents is the exact one builders hit: the skill deferred a real hole, only a passing chat line and an external note remembered it, and the next run opened *clean* as if nothing were outstanding — so the hole sat open for weeks. The ledger in the repo is what makes "come back and finish it" actually work: **write it on the way out (completion rule), read it on the way in (here).** A deferral that lives only in conversation is a deferral that will be forgotten.

## Do these two things in your very first reply — before any analysis

The builder ran this skill because they're nervous. The first thing they see has to make them feel that the right person just took the wheel. So in your **first message**, before you inspect anything:

**1. State the promise plainly.** Tell them exactly what this skill guarantees, in their words — not a vague "let's make things safer." Say it like this:

> *"By the time this is done, three things will be true. (1) You'll have a saved working version of your app you can return to in one command — so you can never lose your work. (2) Every secret that could let someone into your backend will be out of your code, replaced with a fresh one, and tucked into one safe place you control — so nothing leaks and you always know where your keys are. (3) Your data will be backed up automatically, and I'll prove the backup actually restores — so you can never lose your data. After this, every change you make is reversible. I'll walk you through all of it and you'll see each piece get checked off."*

**Then settle the nerves and set the clock — in the same breath as the promise.** The builder arrived scared; say the calming thing plainly and tell them how long this takes, so the length never reads as something going wrong: *"You don't need to be nervous about this — nothing we do here can break your live app; we're only adding safety underneath it, and I do the typing while you watch. It's about 20–30 minutes for a simple app, or closer to 45–60 if you've got a real database we need to back up and restore-test — mostly me working and you confirming a couple of things. There's nothing you can get wrong."* Tune the estimate to what you actually find (a half-set-up project is shorter), but always give one.

**2. Actually open the visible checklist — don't just promise one.** Create the tracked task list *now*, in this first message, with every step on it (see "Set the frame" below for what goes on it). **Saying "I'll set up a checklist" without a checklist actually appearing is the specific failure builders have called out, repeatedly — and it has still happened on real runs even with this warning here. Do not write the sentence "here's the checklist" unless the checklist is literally in the same message.** Render it as an actual checklist the builder can see — checkboxes with every step listed — not a prose description of one. If a rendered task-list tool is available, call it *before* you write any prose claiming a list exists. If not, paste a real markdown checklist (`- [ ] Step …`) into this message and re-paste the full updated version every time anything changes. A reference to a checklist that isn't on screen is the failure; the artifact has to be on screen.

**Keep the displayed step honest — it must match the step you're actually on.** Builders have caught the checklist lagging reality ("it says Step 4 but I'm pretty sure I'm already past it"). So: exactly one item is "in progress" at a time, flip an item to done only the moment it's actually verified, and re-render the list on every transition. If you're doing the work for Step 4, the list shows Step 4 in progress — never still showing Step 3, never already showing Step 5. A checklist that drifts from reality is worse than none, because the builder trusts it to know where they are.

This skill does **not** set up the staging sandbox (the practice copy you preview changes in). That's deferred to `/ship-change`'s first run — a builder doesn't need a sandbox until their first change, and bundling it here turns a 20-minute floor into a multi-hour marathon they bail on. Stay in scope: **git + secrets (out of code, into one safe home) + backups + restore test.**

## Who you're talking to

The builder is **not a developer.** They shipped a real app with AI and it works, but they don't have dev vocabulary. Honor that on every line:

- **Speak builder, not developer.** Never use bare jargon. Say "a saved working version you can always return to," not "a `last-known-good` tag." Always write "safety net," never the bare word "net." Internal mechanics keep their real names; what the *builder reads* must be plain.
- **Gloss every term on first use.** branch, tag, env var, restore, staging, remote — each gets one plain-English sentence the first time it appears. If they hit a word they can't define, you've already failed them.
- **Guided checklist, not autopilot.** Each step: explain what's about to happen and why → do it → let them eyeball it → move on. They should finish *understanding what's in place and able to operate it.* If you silently configure everything, they're left with infrastructure they can't run.
- **Skip what's done.** Git already initialized? Verify and move on. Backups already running? Just confirm them. Don't redo work.
- **Name gaps calmly, not catastrophically.** When you find a hole, the builder is already nervous — don't pour on dread. Lead with what *is* fine, state the gap plainly once, then immediately pair it with the plan to close it — which is to close it *now*, in this run. "Right now this one file has no copy — if it were wiped there's nothing" lands as fear; the same fact lands as *control* when you frame it as a thing you're about to fix: "Here's the one gap left, and here's me closing it right now." Honesty without theatrics.
- **Offer to do everything — never decide for them what to skip.** This is the rule the builder called out hardest, so hold it strictly. Do **not** use a circumstance of their business — "you're closed for the summer," "you have no users yet," "traffic is low," "it's just a side project" — as a reason to skip, defer, or shrink a step. That is you quietly deciding what they want, and it leaves them with less of a safety net than they came for. The default is **every layer, fully, in this run.** A circumstance like "closed for summer" may change *urgency* (it's calmer, not a fire drill) but it never changes *scope*. If a step genuinely can't be completed right now (a credential they don't have on hand, a host with no backup feature), don't silently drop it — lay out the real options and let *them* choose to defer it, on the record, with a date. The builder decides what to skip. You never decide it for them.

## The honest constraint (read before you start)

Much of this work happens inside dashboards you can't see — you can't click Supabase's UI or read Vercel's env panel. So your real role on those steps is **narrator + verifier**: tell them the exact click-path, then confirm the result against something concrete they paste back (CLI output, a backup filename + size, a row count). Any step whose "verify" can't produce such an artifact is a weak point — flag it to the builder rather than pretending it's checked. Never report a dashboard step as done on faith.

**Never push to production *without the builder's explicit okay* — but once they give it, you do the pushing.** Going live is the one irreversible act in this whole skill, and it must never happen without the builder's go-ahead. But read the rule the right way: it means **never push silently or unprompted** — it does **not** mean "refuse to run the push and hand the builder a `git push` command to paste into their own terminal." Handing them the command is the failure: it dumps the scariest, most jargon-heavy step on the least-equipped person. The correct shape is **ask → explain what's going live → and the instant they say yes, run it yourself.** A push *blocked pending confirmation* is correct (pause and wait); a push *you execute the moment they okay it* is correct; a `git push` you paste for them to run is the failure. Keep what you tell the builder short and plain: *"Your change is saved safely on your side. I need your okay before it goes live to your users — just say the word and I'll publish it."* No talk of "local main" or "deploys" — just *saved, and I need your approval to publish it.* Treat every auto-deploy as gated on the builder saying yes, and on yes, you're the one who pushes.

**But don't leave "I haven't published yet" as a surprise at the end — ask at each checkpoint.** Never pushing on your own does *not* mean staying silent about it until the final message. After each significant fix that changes the live code — rotating a secret, removing a hardcoded fallback, adding the rollback command — stop and ask the builder whether they want to publish it, and tell them in plain English exactly what publishing does: *"I've rotated your backend key and the code now reads the new one. This change is saved on your side but not live yet. Publishing it means pushing to your main branch, which auto-deploys to your real app and users in a couple of minutes. Want me to publish it now, or hold it?"* Then track that decision on the checklist (published / holding) so the builder always knows which fixes are live and which are still waiting. The builder discovering at the very end that nothing was published is the failure to avoid — surface push state continuously, not once.

**When verifying needs the builder's hands, hand them one clean command — that's fine.** "You do the heavy lifting, they just watch" holds for most steps, but triggering a backup or restore often needs a credential only the builder has (an admin password, a logged-in session). When it does, don't contort to avoid it: give them a single copy-paste plain-language command, tell them exactly what good output looks like, and verify against what they paste back. One builder-run command with a clear expected result is not a failure of the "guided" promise — an *unexplained* command is.

**But credentials are the hard exception to "they just watch" — keep the secret out of the chat.** A database password or service-role key is the one thing that must *not* flow through the transcript. The clean pattern: the builder pastes the secret **directly into the GitHub Actions secrets UI (or the host's env settings) themselves** — narrate that click-path, and you never receive the value at all. Build the command to read the secret from an env var / secret reference, never as a literal you asked them to type into chat. Be explicit with the builder about *why*: a transcript is a copy of the secret you don't control. And name the trap honestly — if the only way to get a credential is "reset the database password and paste the new one," that paste **lands the live secret in the transcript**, so it's now compromised: have them reset it once *more* after the setup works, or rotate it, and say so plainly rather than pretending the paste was harmless.

## The mental model (show this to the builder)

The end state — built by this skill plus `/ship-change` later — is **two versions of the app with a backup layer under both:**

```
        ┌─────────────────────┐
        │  Your sandbox       │ ← stood up by /ship-change on your first change
        └──────────┬──────────┘
                   │ promote when you like it
                   ▼
        ┌─────────────────────┐
        │  Your live app      │ ← THIS skill hardens (git + secrets)
        └──────────┬──────────┘
                   │ nightly snapshot
                   ▼
        ┌─────────────────────┐
        │  Backups + restore  │ ← THIS skill builds
        └─────────────────────┘
```

- **Live** = the published version users see. This skill hardens it: clean git history, a forever restore point, secrets out of the code.
- **Backups** = a separate safety net *underneath* live. The sandbox/live split protects against *bad code changes*; backups protect against *data disasters that have nothing to do with code* — a user deletes their account, a hack wipes a table, a migration corrupts data. This skill builds it **and tests the restore**, because a backup you've never restored from is not a backup.
- **Your sandbox** = the copy you change things in before they go live. Not needed until the first change, so `/ship-change` stands it up then.

Tell the builder plainly: **"I have a sandbox" does not mean "my data is safe."** It isn't, until backups + a tested restore exist — which is exactly the floor this skill lays.

**The north star for data: every piece of data the app holds is preserved unless a user explicitly deletes it.** "Data" is three layers — database rows, uploaded files/photos, and login accounts — and they live in three different places. The floor is **not done while any one layer is uncovered.** A backup that saves the database but not the user photos is a half-floor, and saying "done" over it is the false comfort this skill exists to kill (see the completion rule at the end).

# Set the frame before you touch anything

Do these three things *before* Step 0, or the builder gets anxious halfway through and bails:

- **Open a visible checklist they watch fill in — and never let an item silently drop.** The builder may struggle with coding, with organization, even with following what Claude is doing. So the single most important thing is that they can always *see* what's happening. Lay every step out as a tracked checklist at the very start (use the task list — it renders as checkboxes they can see). You do the work and take the wheel — that's exactly what they want — but as you go, mark each item in-progress → done so "what's finished" and "what's left" is visible at every moment. **Treat the checklist as a contract: nothing gets closed until it's actually done, and nothing gets dropped.** If something can't be finished now, it does not vanish — it stays on the list, visibly, as a written follow-up they can see (per the completion rule at the end). The failure to avoid at all costs is quietly leaving a needed thing unfinished and moving on — *they will not catch it, so the checklist has to.*
- **Tell them how long it'll take — calibrated to their stack.** A static site or a tiny SQLite app is roughly **20–30 minutes**; a Postgres/Supabase app with real data and a heavier restore test can run **45–60**. Say the number up front so being 30 minutes in doesn't feel like something went wrong.
- **Warn that the auth snags come first.** The slowest part is usually *not* the real work — it's identity: the wrong GitHub account is signed in, the host's CLI is logged out, a deploy gets blocked pending your okay. Expect these, clear them before the real steps, and don't let them read as "the skill is broken." They're the toll booth, not the road.
- **Investigate before you recommend any change the builder can't easily undo — never reflex-suggest a setting flip.** A confident recommendation that turns out to be wrong (or breaks their live app) costs more trust than a slower, checked one. Before you propose flipping a setting, changing a config, or removing something — especially anything touching how the app is hosted, deployed, or made public — **first dig into *why* it's the way it is.** A thing that looks like a mistake is often load-bearing: a repo is public because a free hosting tier needs it, a key is in the browser because it's a public-by-design anon key, a branch exists because a deploy watches it. Check the host config, `stack.md`, and the actual dependency before you open your mouth. If you find a real reason, *that's the answer* — say it plainly and leave it alone. The pattern to kill: surfacing a "problem" and proposing the fix in the same breath, before you've confirmed it's even a problem. (The public-repo case in Step 1 is the canonical example.)

# The steps

## Pre-flight — Lock onto the real, live copy of the app (do this before any step below)

**Before you read or touch a single file, decide *which copy of the app you're looking at* — and refuse to work from the wrong one.** This is the most damaging failure in the whole system, and it's silent. A builder usually has several copies of their app on disk: an old `~/Documents/your-app` clone, a copy synced in iCloud/Google Drive, a fresh `git clone` from last week. They look identical in a file browser. But these skills write durable state files (`stack.md`, `readiness-check.md`, `safety-net.md`, …) **into the repo**, so if you read or write the *wrong* copy, every downstream symptom follows: a re-run "forgets" what a prior run did (its file lives in the other copy), one skill swears another never ran, the builder is told "you have no backups" when their real repo has a full safety net — and, worst of all, you can overwrite weeks of their real work with a stale version. For a skill that *sets up the safety net*, this is fatal: building a restore point and backups on a stale copy gives the builder false protection over the wrong app. **This has actually happened: a skill worked from a copy that was 61 commits behind the builder's live app.** Never let it happen again.

**Resolve the canonical copy, in this order, before anything else:**

1. **If `stack.md` records a canonical path, trust it first.** `/readiness-check` writes `canonical_repo:` (the absolute path of the real copy) and `canonical_remote:` (the GitHub URL) into `stack.md`. If you're not already inside `canonical_repo`, switch to it before doing anything. If you can't even find `stack.md` where you are, treat that as a strong sign you're in the wrong copy — go looking for the others before concluding none exists.

2. **If there's no recorded path, find every copy and pick the live one.** Search the common roots (Documents, Desktop, the user's cloud-drive folders) for clones of this app. For each, read its remote (`git remote -v`) and how current it is. **The live copy is the one whose `main` matches the GitHub remote's `main` — not whichever folder this session happened to open in.** If the builder has told you which copy is real (e.g. "use the Drive one"), honor that and verify it.

3. **Always compare local to the remote before trusting any copy.** Run `git fetch`, then check whether local is behind origin (`git rev-list --left-right --count main...origin/main`, or `git status`). **If the copy you're in is behind the remote, STOP.** Do not read its state files as truth, and do not write to it. Switch to a copy that's current, or — with the builder's okay — pull/clone fresh. A copy that's 61 commits behind is not "the app"; it's a photograph of the app from weeks ago. **The GitHub remote is the source of truth about what exists** — when you're unsure whether a net is already in place (a backup workflow, a restore point, a safety-net manual, a backups repo), check the live remote, because a prior pass may have set up things your local copy never saw. Never tell a builder "you don't have X" — or rebuild an X they already have — from a local copy you haven't confirmed is current.

4. **Record the decision so no future run repeats this.** Once you've confirmed the live copy, make sure `stack.md` carries `canonical_repo:` and `canonical_remote:` (write them if missing) and commit it — so every later skill and every re-run locks onto the same copy instead of re-guessing. This is plumbing: do it silently, never make the builder adjudicate which folder is real.

## Step 0 — Read the stack (or discover it if `/readiness-check` didn't)

**`/readiness-check` owns `stack.md` now — your job here is to *read* it, not write it.** If the builder ran the front door first (the normal road: `/readiness-check` → `/safety-net` → `/ship-change`), `stack.md` already records the stack, the project type, and the platform case. Read it, confirm it back in one plain sentence (*"your notes say this is a Supabase + Vercel marketplace — still right?"*), and move on. Do **not** regenerate it — two skills writing the same file is how a builder ends up with two conflicting stack fingerprints.

**If `stack.md` is missing** (the builder came straight here without the front door), do two things: (1) gently note that `/readiness-check` is the front door and worth a 5-minute run sometime — *recommend it, don't block on it*; and (2) discover the stack yourself as a fallback so this run isn't stuck. Ask in plain English: *"What's your frontend? Your backend? Your database? Where's it hosted?"* If they don't know (common), inspect the repo and propose an answer to confirm: *"Looks like Next.js, Supabase for the database, and Vercel for hosting — does that sound right?"* Write `stack.md` yourself in that case, so downstream skills still have it.

Either way, confirm **what kind of project this is** — it decides which steps even apply. A static site has no database; running a "restore test" on it is nonsense.

| Project type | What it looks like | Which steps apply |
|---|---|---|
| **Static site** | HTML/CSS/JS, no backend, no DB | Git + secrets sweep. Skip backups/restore — git history *is* the backup. |
| **App with backend, no users yet** | Has a DB but no real user data | Git + secrets + DB backups + restore test. |
| **App with real users / payments** | Real people, real data | All of the above; the data choices in `/ship-change` matter most here. |
| **Mobile app (Expo / React Native)** | Phone app + web build | All of the above + the client-side "secrets" rule in Step 2 is critical. |

**Also ask where they *built* it, not just where it's hosted — an all-in-one builder platform changes this skill's job.** Replit, Lovable, Bolt, v0 and similar already auto-snapshot code *and* data after every change, give one-click rollback, and ship a built-in secrets pane. If the builder is on one of these, don't fight it or duplicate it — make the platform's built-in checkpoint/rollback their primary "way back" (narrate it, have them watch it work once). Then aim this skill's real effort at the three things those platforms *don't* do: **rotating** an already-leaked key (the pane hides a value, it doesn't revoke a key that's already sitting in git history), a **portable backup that survives leaving the platform** (their snapshots vanish the day they export), and the **teaching** so they can operate it themselves. On a raw stack — a real git repo on GitHub plus a host like Vercel/Railway plus a database like Supabase — run every step as written; this is where the full floor matters most and where nothing else is covering them. (Hybrids exist: built on Lovable but pushing to a real GitHub repo → treat it as a raw stack.)

**The platform case (all-in-one builder vs. raw stack) lives in `stack.md`** — `/readiness-check` records it. Read it from there. If you had to discover the stack yourself (the fallback above, when `stack.md` was missing), then *you* write it, including the platform case. Either way, every later step relies on it being right.

## Step 1 — Git baseline

> *Plain-English gloss to give them:* git is the system that remembers every version of your code so you can always go back. A "branch" is a separate copy you work on without touching the live one. A "remote" is the online backup of your code (usually GitHub).

- If there's no git repo, **init one.**
- Write a `.gitignore` that excludes `.env`, secrets, and platform junk — **do this before the first push, not after.**
- **Sequencing rule — never push to a remote before the Step 2 secrets sweep has run.** Pushing first and cleaning up second *creates* the exact leak this skill exists to prevent: once a secret hits a remote it's compromised whether the repo is public or private. So on a project with no remote yet, run Step 2's scan, fix `.gitignore`, rotate anything sensitive, *then* push. (The step order below assumes an app already on GitHub; for a never-pushed repo, do Step 2 before the first push.)
- If there's no remote, push to a **private** GitHub repo.
- **If the repo is already public: investigate *why* before you breathe a word about flipping it.** Public is frequently load-bearing, and recommending "flip it to private?" before you've checked is the kind of confident-but-wrong move that destroys the builder's trust — and can break their live app. **Dig first, in this order:** (1) check what depends on the repo being public — a free deploy/hosting tier that won't build private repos, GitHub Pages on a free account, a Supabase/Vercel/Netlify free plan wired to public-repo deploys, anything in the host config or `stack.md` that reads "free tier"; (2) run Step 2's secrets sweep so you actually know whether anything sensitive is exposed. **Only after both** do you say anything — and shape it to what you found: if public is load-bearing (a free tier needs it) **do not recommend flipping it** — say so plainly (*"your repo's public because your free hosting plan needs it to be — that's a real reason, so we leave it public"*), make "no admin secret in source, ever" a permanent rule, and lean harder on Step 2's rotation. Only if public is *not* load-bearing **and** you've confirmed it serves no purpose do you offer the flip — and even then as a question, not an order. **Never lead with "flip it to private?" as a reflex.** "Nothing secret leaked AND there's a deliberate reason it's public" must stop the recommendation cold.
- **Tag the current commit as a forever restore point.** Call it that to the builder — "a saved working version you can always return to in one command." (Internally the git tag is `last-known-good`; in the plain-English version log `/ship-change` keeps, this launch point is labeled `v1`.)
- **Name the difference from Claude's `/rewind`, because the builder probably thinks that's already their safety net — it isn't.** If they use Claude Code they've seen `/rewind` (a.k.a. `/checkpoint`) undo recent edits, and they may believe their work is protected. Say it plainly: `/rewind` is a *within-the-session* undo — it's wiped after 30 days, it's gone the moment they close the session and open a new one, and it doesn't track files changed by commands (a `DROP TABLE`, an `rm`, a `mv` is permanent to it). The same is true of a platform's "undo last message." What you're tagging instead is a *permanent* saved version that lives in the code's own history and survives forever. The session undo is a handy convenience; **the restore point is the actual net** — don't let the convenience masquerade as the net.
- **Give them a way back they can actually use in a panic — and a typed terminal command is NOT it.** A non-technical builder cannot, mid-emergency, open a terminal, navigate to the right folder, and type `./rollback.sh` correctly. Treating that as "the whole command, that's it" — builders have flatly rejected it. The way back has to survive someone who has never opened a terminal. Give them, in this order of preference:
  1. **The host's one-click rollback, if it has one — this is the real answer for this audience.** Vercel, Netlify, Render and similar keep every past deploy and let you click an older one → "Promote to Production" / "Instant Rollback" — going back is a *button in the dashboard they already log into*, no terminal at all. If the host has this, make it their primary way back: narrate the exact click-path, have them do it once on a past deploy to *see* it work, and write that click-path (not a shell command) as line one of the cheat sheet.
  2. **"Just tell your AI: roll me back to the saved working version."** Their everyday way back is to *say the words* to you (or whatever assistant they use) and have it run the mechanism. Frame the restore point as something the AI executes on request — the builder remembers a sentence, not a command.
  3. **The named script as the mechanism underneath, not the user-facing instruction.** Still *create* a tiny script behind a plain name (`npm run rollback`, `./rollback.sh`, a Makefile target) that does the real work — it's what the AI runs in (2) and a backstop if the host has no button. **Get the tag-deref right inside that script:** when it force-pushes the restore point to the deploy branch, **dereference the tag to its commit** (`last-known-good^{commit}`), because if `last-known-good` is an *annotated* tag, pushing `last-known-good:main` puts the *tag object* on the branch instead of the commit — a real bug a prior emergency drill caught. But never hand the builder a raw `git push --force origin last-known-good:main`; they will not run it correctly while panicking, and force-pushing live is when stress is highest. The script restores to the saved working version locally and then **stops and asks before anything goes to production** (per the never-auto-deploy rule above). In the cheat sheet, the script name is the *fallback* line, below the dashboard click-path and the say-the-words option — not the headline.
- Set up a workflow where changes reach the live version through a **separate branch, not by editing live directly.** For a solo builder on a private repo, GitHub's formal "branch protection" toggle is mostly ceremony — you're the only committer. The real win is the *habit*: a working branch you change things on, the main branch reserved for what's live. If they want the hard guardrail anyway, narrate the click-path to enable branch protection in the GitHub UI and confirm it from the settings page — you can't toggle it yourself.

## Step 2 — Secrets out of code

> *Gloss:* an "env var" (environment variable) is a secret value — like a password or API key — stored in your host's settings instead of written into your code, so it never ends up in your public code history.

- **Run a real secrets scanner as the net, then read by hand for judgment — grep alone misses things.** A by-hand grep catches known shapes but misses high-entropy secrets that don't match a familiar prefix, and it only sees the *current* code — a key you already deleted from a file is still live in an old commit. The mature scanners close both gaps: `gitleaks detect` (or trufflehog) uses entropy analysis plus a provider-pattern library *and* walks the whole **git history**. So run one over the repo and its history as the exhaustive sweep. But the scanner only finds *strings that look like secrets* — it can't tell an admin key from a public-by-design one, and that judgment is the part that matters. So still **open and read every auth, config, settings, and environment file yourself**, and grep the entire tree for the shapes below as a backstop. The scanner is the net; your reading is the decision. The patterns to hunt for, all of them:
  - API keys, tokens, passwords, connection strings, private keys, webhook secrets.
  - **Signing / session secrets** — `SECRET_KEY`, `JWT_SECRET`, `SESSION_SECRET`, anything that signs login tokens or cookies. These are the most dangerous of all: if the value is guessable, an attacker can *forge a login token and become any user*, including an admin. Treat every signing secret as admin-tier (bucket 2 below).
  - **Env vars with a hardcoded fallback default** — `os.getenv("SECRET_KEY", "some-default")`, `process.env.X || "default"`, `process.env.X ?? "default"`, `ENV.fetch("X", "default")`. **Reading a value from an env var does NOT make it safe.** If there's a literal fallback baked into the code, that fallback is a committed secret *and* it's whatever production silently runs on if the env var was never set on the host. This is exactly the kind that gets missed because the line "looks like it uses an env var." It does not get a pass. For every such fallback: flag it, treat the value by its bucket below, **and confirm the real env var is actually set in production** (an unset env var means the app is running on the guessable default right now).
- When the scan is done, tell the builder you checked the whole codebase, and list what you found bucketed as below — so they can see nothing was skimmed.
- **The rule is NOT "move every key to an env var."** Some keys are *designed* to be public and ship in the browser on purpose — a Supabase anon key, a PostHog key, a Sentry DSN. Sort every key found into three buckets:
  1. **Public-by-design → leave it.** Anon keys, analytics keys, error-reporting DSNs. **One caveat that matters for the Supabase/Firebase anon key: it's only safe to ship if the database's row-level security is ON.** The anon key is *meant* for the browser — but with row-level security off (the forgotten default), it lets anyone read or write your whole database straight from the browser console. This skill doesn't audit that (it's `/go-live-check`'s wide-open-database check) — **but a Gate-1-only builder never reaches that check**, so don't bless the anon key with a flat "fine to ship": say it's fine *because it's public by design — as long as row-level security is on*, and if you can't confirm it's on here, drop a line in the `## Still open` ledger to verify it.
  2. **Admin / write access → move it AND rotate it.** Anything that grants write/admin access to the backend — a database service-role key, a payment secret key — must never sit in the browser bundle or a public repo. It lives only in server-side env vars or an edge function.
  3. **Public-but-billable → leave it in code, but lock it down at the provider.** A metered client key (maps, places, geocoding) is *meant* to ship in the bundle, but if it isn't restricted at the provider (by referrer/app + allowed APIs) anyone can lift it and run up your bill. This is a *money* leak, not a data leak. Don't wave it through — have them add the provider restriction.
- **Rotate before you relocate — moving a leaked key does not un-leak it.** An admin/write key that was ever committed is compromised: it still sits in git history, and on a public repo anyone who saw it has a working key *right now.* Moving it to an env var changes nothing for them. So for every sensitive key that touched the repo: **generate a new one, revoke the old one, then put the new one in env vars.** Relocating without rotating leaves the builder *feeling* safe with a live, stolen backend key — the exact false comfort this skill exists to kill. (Scrubbing the key from old commits is a later polish; rotation is what closes the hole — do it first and always.)
- **A guessable signing/admin secret is an *active live exploit*, not a "layer to schedule" — it is the hardest thing in this skill to defer.** A committed-or-guessable `SECRET_KEY`/`JWT_SECRET` with a hardcoded fallback (the `os.getenv("SECRET_KEY", "dev-secret")`-on-line-15 case) means *anyone can forge a login token and become an admin on the live app right now* — it is being exploited the moment someone notices, not a someday risk. So unlike a data layer the builder may knowingly choose to defer, this one you **push hard to close in this run.** If it genuinely cannot be closed now (a credential they don't have on hand, a deploy they can't approve yet), it does not become a quiet footnote — it goes to the **top of the `## Still open` ledger flagged 🔴 ACTIVE-EXPLOIT**, with the file and line, and you re-surface it loudly at the very start of the next run (per the return-trip section). Never let a forge-a-login hole leave the session remembered only in chat. **Keep this heat internal, though — it's here to stop *you* waving the hole through, not to frighten the builder.** To them it stays calm, per "name gaps calmly": *"there's one important thing worth fixing right now — a key that could let someone into your accounts — and I can close it in a few minutes,"* never *"you're being hacked as we speak."*
- **Confirm the app still runs — without a risky live deploy.** Sequencing trap: a vibe-coded app often only "runs" in production, and the staging sandbox doesn't exist yet (it's deferred to `/ship-change`). So "just deploy and see" would push the secret-move straight to live — the unsafe act this whole skill is built to prevent. Prefer a **local run** if the project can run locally. If it genuinely can't, treat this as the one moment that needs care: **change one key, deploy, verify, repeat** — never move every secret and deploy blind. Flag this to the builder rather than glossing it.
- **The ordering trick that hardens a secret without a scary blind deploy.** Naming the real tension: fixing a leaked admin key means a code change *and* a production deploy — but the staging sandbox doesn't exist yet (it's deferred to `/ship-change`), so you can't preview it safely. The skill that protects them seems to force the one risky act it warns against. The way through is **ordering**: (1) add the new, rotated key as an env var and make the code read the env var *first, falling back to the old value* → deploy → confirm it still runs; (2) only now remove the old hardcoded value → deploy again. The app is never in a broken state on live, and you never deploy blind. Name this sequence to the builder — it's the move that dissolves the tension instead of gambling on it.
- **Make sure it can't silently happen again — two layers, both cheap, and this is the part most secret-cleanups skip.** Finding and rotating today's leaked keys closes today's hole; it does nothing about next week, when the builder's AI helpfully pastes a fresh API key straight into the code again. The builder won't catch it — so close the loop mechanically:
  1. **Write the rule into the project's AI-memory file, so the assistant that caused the leak is told, permanently, not to.** The builder works *through* an AI, so the durable fix is an instruction that AI re-reads every session. Append a short explicit rule to whatever memory file their assistant uses — `CLAUDE.md` (Claude Code), `AGENTS.md`, `.cursorrules` (Cursor), or the equivalent: *"Never hardcode secrets, API keys, tokens, passwords, or connection strings in source — they live only in environment variables / the host's secrets settings. Never add an env var with a hardcoded fallback default (`getenv("X","default")`, `process.env.X || "..."`). Public-by-design keys (Supabase anon, PostHog, Sentry DSN) are fine in client code."* If several memory files exist, write it to the one their assistant actually reads; if none exists, create the right one for their tool.
  2. **Install a `gitleaks` pre-commit hook, so a secret literally *cannot* be committed.** The written rule is guidance; the hook is enforcement — it scans the staged change and **blocks the commit** the instant a secret appears, no matter which tool or which person wrote the code. Set up a pre-commit hook that runs gitleaks against the **staged** change — but **don't hardcode the subcommand, because it changed:** `gitleaks protect --staged` was the old form and is **deprecated since v8.19** (still hidden-but-present for now, gone eventually); current gitleaks uses `gitleaks git --staged` (the `git` subcommand with `--staged` scans the pre-commit set). So **check the installed version and use the invocation it supports** — run `gitleaks version` / glance at `gitleaks git --help`, use `gitleaks git --staged` on v8.19+ and fall back to `gitleaks protect --staged` only on an older binary — or wire the project's official gitleaks pre-commit hook, which tracks this for you. Commit the hook config so it travels with the repo, and **prove it to the builder**: try to commit a fake key and let them watch it get rejected. (The proof is the real gate — a wrong or rotted command fails *visibly* here, before it can give false comfort.) This is the one mechanical guard worth the few minutes of setup — it turns "please remember not to" into "you can't." (This is also what protects every future `/ship-change`: a leaked key in a new feature gets stopped at commit time, before it can reach the repo.)

- **Give the keys one safe home — a password manager — so they don't live in the builder's head, a notes file, or one laptop's browser.** Getting secrets *out of the code* is only half the job: the builder still has a pile of things they must keep *somewhere* — the logins to every dashboard (the host, the database, the payment provider, GitHub, the company they bought the web address from), the fresh keys you just rotated, and any recovery codes. Right now those usually live in the worst places (memory, a Notes file, a sticky, autofill on a single laptop). A password manager is the one safe, portable home for all of it — **and it's the tool the top-of-the-ladder `/handoff` step later assumes already exists**, so it belongs here at the floor, not years later.
  - > *Gloss:* a **password manager** is a single locked app (1Password, Bitwarden, or the one already built into your browser / Apple / Google) that remembers all your logins behind one master password, so you don't have to.
  - **Already have one?** (browser built-in, Apple Keychain, Google Password Manager, 1Password, Bitwarden) — don't make them switch. Confirm it, and make sure the app's key dashboard logins are actually in it.
  - **Don't have one?** Set one up — the built-in browser/Apple/Google manager is a fine, free start for this audience; don't oversell a paid tool. You narrate the click-path; **the builder creates it and holds the master password** — that one thing stays only in their head, you never see it or store it.
  - **All-in-one builder platforms** (Lovable, Replit, v0) already ship a built-in secrets/teammate panel — if that's their whole world, that panel can *be* the keys-home; don't send them to install a separate consumer tool on top of it.
  - **Map-not-vault holds here too:** you record in `safety-net.md` that a password manager *exists* and *which one* — **never a single password value.** The manager holds the secrets; the cheat sheet only points at it.
  - **Put the freshly-rotated keys straight into it** as you rotate them (above), so a new value has a home the moment it's created and never goes back into the code or the chat.

## Step 3 — Automated backups
*(Skip for static sites — git history is the backup.)*

- **Audit what already exists before you add anything.** Vibe-coded apps are full of half-built backup code the builder *thinks* is working — a backup routine pointing at the wrong file, writing to a bucket that was wiped, or silently erroring for weeks. A broken backup the builder trusts is more dangerous than no backup, because it removes the fear that would otherwise make them careful. So first: find any existing backup mechanism, and prove it produced a *real, recent, restorable* file before relying on it. If it's broken, fix or replace it — don't stack a fresh backup on top of a rotten one and call it redundancy.
- Turn on scheduled backups on the database. If the host has a built-in toggle (Supabase has it; Firebase exports; most managed Postgres providers), use it — narrate that click-path and let the builder own it.
- **"My host backs up databases" rarely means "my host backs up *this* data."** Managed-database backups only cover the *managed* database. If the app stores data as a plain file on a disk or volume — a **SQLite file on a Railway volume** is the textbook case — the host does **not** back that up, even while it faithfully backs up its managed Postgres/MySQL plugins. The builder reasonably assumes they're covered; they aren't. Name the exact store their data physically lives in and confirm *that specific thing* is what's being backed up.
- **Check where the backups LAND — and that the destination is private.** This is a data leak hiding inside the backup step: a full database dump written to a *public* bucket is worse than no backup — it's your entire dataset, downloadable by anyone who finds the URL. Builders routinely aim backups at the same public R2/S3 bucket they use for everything else. So for every backup destination confirm two things: it's **durable** (not just the builder's laptop — laptop dies, backup dies; and *not committed into the **app's** repo* — a binary dump bloats the code repo and can leak alongside it; a *separate private* backup repo is fine and is covered below), and it's **private**.
- **Where backups should live — give the builder a concrete recommendation, not just criteria.** Pick by stack and say it plainly:
  - **If the host has its own managed backup store** (Supabase scheduled backups / PITR, Firebase scheduled exports, a managed-Postgres provider's automatic backups) — use *that* as the primary home. It's already private, durable, and off the builder's machine, and the host handles retention. **But check the tier:** the common free tiers (Supabase Free is the textbook case) give you **no managed backups at all** — the toggle is paywalled. Don't assume "it's Supabase, so backups are handled"; confirm the tier actually includes them before relying on it.
  - **If the host won't back it up for you (free tier, or a plain file like SQLite) — the realistic free path is a scheduled GitHub Action that dumps the data and commits it to a *dedicated private backup repo*.** This is the answer that actually fits this audience: it's free, it runs on a schedule with zero servers to maintain, the builder sets it up once, and GitHub keeps every dated snapshot for free (the commit history *is* your versioned retention). Name it as the pattern — don't make the builder discover it. A scheduled `pg_dump` cron on a box they own is the thing this audience cannot stand up or maintain; the GitHub Action is its replacement, not an aside.
  - **A dedicated private object-storage bucket** (a new private Cloudflare R2 / AWS S3 / Backblaze B2 bucket, public access off, separate from the bucket the app serves files from) is also a good home, especially for larger or binary data — keeping it separate from the assets bucket prevents the "backup landed in the public assets bucket" leak.
  - **Reconciling the "never git" rule:** the hard nos are still a **public** repo/bucket, the **app's own** repo (a binary dump bloats it and can leak with the code), or **only the builder's laptop**. A *separate, private* backup repo is not those things — it's a legitimate, free, versioned home for modest data, and you should use it when the host gives you nothing. The line isn't "never git," it's "never public, never the app repo, never laptop-only."
  - **Size trip-wire — say it out loud as a watch-item, not a soft aside.** A backup repo is fine for a few hundred MB of database dumps and a modest pile of photos. It is *actively wrong* for a media-heavy app — a growing video or large-image app will blow past git's practical limits and make every clone painfully slow. So when you set up a git-repo backup, tell the builder the explicit trip-wire: **"This is the right call now. Once your uploaded files get large or you start storing video — call it past a few GB — move the *file* backups to a versioned object-storage bucket; the database dumps can stay here."** Write that trip-wire into the cheat sheet as a named future task, so it doesn't get silently outgrown.
- **Don't let a sync quietly become your only copy — keep dated versions.** The data north star ("preserved unless a user deletes it") has a sharp edge: a backup that just *mirrors* the live state — one folder it overwrites each run — will faithfully propagate a disaster. If a hack or bad migration mass-deletes rows and the next backup run mirrors that empty state over your only copy, the backup *erased* the data, not preserved it. The fix is one rule: backups must be **versioned/retained** — multiple dated snapshots (git history gives this for free; object storage needs versioning turned on), not a single overwritten mirror. Then a propagated delete just adds one bad snapshot next to the good ones, and you restore from yesterday.
- **Verify backups are actually landing somewhere**, not just "configured." Have them show you the latest backup file — name, size, timestamp. A configured-but-empty backup is the most common silent failure.
- **Back up user-uploaded files too, not just the database.** Many apps keep photos/documents/uploads in object storage (Supabase Storage, S3, Firebase Storage) — that data is separate from the database and is **not** covered by a DB backup. A builder who loses every user's uploaded photo is as dead as one who lost a table. If Step 0 found a storage bucket, confirm it's backed up (or at minimum versioned/retained by the host) — and say plainly if it isn't covered.
- **Watch for separately-managed auth data.** On managed-auth stacks (Supabase `auth.users`, Firebase Auth), user accounts live *outside* the tables a default dump captures — a `pg_dump` of the `public` schema won't include `auth`. A backup that silently omits auth restores your data and leaves every user locked out. Make sure the backup scope includes auth, and note it explicitly in the cheat sheet.

## Step 4 — Restore test (un-skippable)
*(Skip only for static sites.)*

**A backup you've never restored from is not a backup.** This step is the whole point of the others — keep it a *real restore*, never a file-integrity check standing in for one (checking a file is "non-empty and opens" proves the file exists, not that you can ever get your app back).

> *Plain-English glosses for the words in this step — say each once, the first time it comes up, so the builder never hits a bare term:* a **throwaway database** is a temporary empty copy we restore into and then delete, never your live one; a **row count** is just how many records a table holds; an **extension** (like **PostGIS**) is an add-on a database needs to understand special data such as map locations; **auth roles** are the invisible account-types your login system leans on; **`pg_dump`** is the standard tool that makes the backup file; **Docker** is a background tool that lets a database run on your own computer (you never type a Docker command — I handle it). None of these are things the builder has to learn; they're just named so a word never lands cold.

- Take the most recent backup.
- Restore it into a **throwaway empty database (NOT live).**
- Confirm the restored data looks right — row counts, a few sample records — **and that user accounts came back too** if the stack has separately-managed auth (Step 3). A restore that brings back the tables but not the logins isn't a passing restore.
- **Compare per-table counts against the *source*, and treat a table that restored its schema but *zero rows* as a FAIL to investigate — not a pass.** "Row counts look right" is a trap when the only thing you eyeball is the restored copy: a table that came back with its structure but **none of its data** shows `0` and scrolls past as if that's fine. So don't just glance at the restored numbers — **line them up against the live source you backed up from.** A table the source says holds 4,000 rows and the restore says holds 0 is a **silent partial-restore failure** (a broken export, a missing extension/role the rows depended on) — flag it and diagnose, the same way the "13 of 63 tables" case below is handled, except here the table is *present but empty* rather than missing, which an eyeball of the restore alone never catches. (A table that's *supposed* to be empty is fine — the flag is specifically source-says-N-rows, restore-says-0.)
- **The restore target must match what the backup needs — this is where restores silently half-fail.** A managed-database dump is *not* a self-contained file you can pipe into any empty database. A Supabase dump in particular assumes the environment it came from: the `extensions` schema, any Postgres **extensions the schema depends on** (PostGIS is the trap — any `geography`/`geometry` column, like a map-location field, needs it), and the **auth roles** (`anon`, `authenticated`, `service_role`, etc.). Pipe that into a bare local Postgres and you get a *partial* restore: most tables load, but every table that needs a missing extension or role fails to build. The symptom is the dangerous part — "restored 13 of 63 tables" or "every table came back except the one with the map column." **Do not read a partial restore as either a false fail ("your backup is broken!") or a false pass ("counts look fine, done!"). It almost always means the restore *environment* was missing a piece, not that the backup is bad.** Diagnose which extension/role/schema was missing and supply it.
- **So choose the restore target deliberately — and for managed stacks, native is both correct and fast.** The restore *method follows the database recorded in `stack.md`*; the options below are instances of the pattern. (Firestore, Mongo, MySQL each restore differently — use the right tool for whatever Step 0 recorded.)
  - *Managed-host native — this is the default for Supabase/Firebase/managed Postgres; reach for it first.* Restore into a **fresh throwaway project of the same kind** (a new free Supabase project, a Supabase branch, the host's "restore to a new instance"). It already has the extensions, PostGIS, auth roles, and `extensions` schema the dump expects, so the restore is *clean and fast* — no installing dependencies. This is the path that avoids the time-sink below. Creating the throwaway project is a host action, so it's gated on the builder's okay (per the never-act-on-their-host-without-asking rule) and needs them logged in; if the free tier caps projects, have them delete an unused one or fall back to the local path.
  - **Tear the throwaway down the moment the counts check out — this is not optional.** That throwaway project is now holding a full, real copy of the builder's user data, including logins. A forgotten throwaway is a second live database of everyone's data sitting unguarded — the exact data-leak this skill exists to prevent, created *by the safety check itself*. So the sequence is restore → eyeball counts → **delete the project**, and you confirm the deletion out loud ("the throwaway copy is now deleted — your data isn't sitting anywhere it shouldn't") rather than leaving it as a step the builder might skip. **Prove it's actually gone, don't just click delete:** reload the host's project list and confirm the throwaway no longer appears (or that its URL/endpoint now 404s) — a "delete" that's really a soft-delete, or a teardown still in progress, leaves a full copy of everyone's data live. The builder witnesses the empty list; proof-not-presence applies to the cleanup too. Put nothing real on a throwaway you don't immediately destroy.
  - *Local throwaway — only if you pre-seed the environment first.* A temporary Postgres (Docker, or a `brew`-installed local Postgres when there's no Docker — name the no-Docker path, this audience often has neither) works *only* if you first create the `extensions` schema, install the required extensions (detect PostGIS from the schema — grep the dump for `geography`/`geometry`/`postgis` before you start), and create the auth roles. Skip that pre-seed and you'll hit the partial restore above. **Heads-up on speed:** installing PostGIS into a bare local DB pulls a chain of dependencies and can take several minutes — this is the single slowest thing in the whole skill, and it's avoidable by using the managed-native path instead. If you're already mid-install, tell the builder *that's* what's slow ("installing the map extension, it pulls a few dependencies"), not "something's stuck."
- **Effort scales with the database — set that expectation.** A SQLite file restores in seconds; a Postgres/Supabase dump with real volume takes real minutes, and a local restore that has to install an extension takes more. Tell the builder which case they're in up front so a slow restore reads as "big database / installing a dependency," not "something broke."
- The builder's job is to confirm the numbers look right; you run the steps. Done this way it's minutes, not a marathon — which is what keeps it un-skippable instead of the step everyone bails on.
- **Leave proof, not a promise — write a committed restore-test record.** The builder's deepest fear is being *told* the backup works and finding out it doesn't on the day they need it. A row count that scrolls past in the chat is not proof — it's gone when the session ends. So capture the result as a file committed to the repo (`restore-test.md`, or a dated section in the cheat sheet) recording: **the date/time of the test, the exact backup it used (file name + size + timestamp, or the snapshot commit), the per-table row counts *checked against the source's counts* (so a present-but-empty table is a caught FAIL, not a passing 0), the auth-account count that came back, PASS/FAIL, and the exact command to reproduce the test.** This is the artifact the builder points at later — durable, dated, and re-runnable on demand. Tell them plainly: *"This isn't me saying it works — here's the dated proof in your repo, and here's the one command that re-checks it any time you want."* The honest upgrade, name it as the next step: have the scheduled backup job **also run this restore-and-count on a schedule** so the proof stays *current* instead of being true only on setup day — a backup proven once six months ago is most of the way back to a promise.

## Step 5 — Cheat sheet

Write `safety-net.md` into the repo, in plain English:
- **How to go back, easiest path first:** the host's one-click rollback click-path (if it has one), then "tell your AI: roll me back to the saved working version," then the named script as the fallback line. Not a bare shell command as the headline.
- Where backups live, the restore command, and **what the backup actually covers** — database, user-uploaded files/storage, and auth accounts — so a future restore doesn't silently miss a layer.
- **A pointer to the dated restore-test proof** (`restore-test.md`) and the one command that re-runs it.
- **The size trip-wire** if backups live in a git repo: when files get large / you add video, move file backups to versioned object storage.
- Which env vars exist, where they're set, and which are server-only vs. safe-to-ship.
- **Where the keys live: the password manager** — which one it is, and that the app's dashboard logins + freshly-rotated keys are kept there. **The name and location only, never a value.** This is the home the `/handoff` packet points at later.
- **The leak-recurrence guards in place:** the "never hardcode secrets" rule written into the project's AI-memory file, and the `gitleaks` pre-commit hook that blocks any future commit containing a secret — so a later change can't quietly re-leak.
- **A `## Still open` ledger — even if it's empty.** This is the durable home for anything deferred: every item the floor didn't close this run, written as a dated, prioritized line the builder can read later (🔴 first), e.g. *"🔴 2026-06-25 — rotate JWT `SECRET_KEY` + remove fallback in `backend/auth.py:15` (forge-able admin login until done)"*, *"2026-06-25 — push drafted `safety-net-followup` branch (rollback.sh + cheat-sheet)"*, *"2026-06-25 — turn on photo versioning on the public R2 bucket."* If nothing was deferred, write the header with *"(nothing outstanding)"* so a future run knows the ledger is real and current, not just missing. This is the exact list the **return-trip section at the top** reads on the next run — it's what turns "we'll get to it later" from a forgotten promise into a tracked one.

Commit it. Future-you and future-AI read this and operate everything without re-deriving.

**Do not declare the floor "done" while any data layer is still uncovered.** The default is that you close every layer *in this run* — don't leave one open because the moment feels low-stakes. If a layer genuinely can't be closed right now, the skill is **not complete** — it's *partially* complete with a named, dated gap **the builder themselves chose to defer**, not one you decided to skip for them. Say exactly that ("your database is safe and proven; your uploaded photos are *not* covered yet — want me to close that now, or do you want to schedule it?") instead of reporting "completed" with the gap buried in a footnote. Per the data north star: every piece of data is preserved unless a user deletes it, so each layer either gets covered now or becomes a written commitment with a date the builder signed off on.

**Every deferral lands in the `## Still open` ledger in `safety-net.md` — a chat message is not a record.** This is the close to the return-trip section at the top: a gap that lives only in the conversation (or in some external note) is a gap the next run will open blind to. So the instant the builder chooses to defer *anything* — a data layer, a secret rotation, an unpushed follow-up branch — write it into the committed ledger as a dated, prioritized line *before* you close the session, and read it back to them: *"I've written these three open items into your repo so we — or you, or any AI you ask next — can't lose track of them; the JWT one is flagged red because it's live until it's fixed."* The end-of-run summary names what's closed **and** points at the ledger for what isn't. A clean-sounding "all done" over an unwritten deferral is the failure this whole loop exists to kill.

Close by telling the builder, in their words, what they now have: *"You can't lose your code — there's a saved version you can return to in one command. You can't lose your data — it's backed up nightly and I've proven the backup actually restores. Secrets that could compromise your backend are out of your code, rotated, and kept in one safe place you control — and there are guards in place so a future change can't quietly leak one again. From here, every change is recoverable."*

## Hand them onto the next step — don't end at a dead stop

The whole system is a road, and the builder should never be left wondering "okay… now what?" The road is `/readiness-check` → `/safety-net` → `/ship-change`, so **the floor being done means the next move is `/ship-change`** — the safe-change loop that lets them actually *use* the net you just built. End every run by offering it directly, as the unmissable last line:

> **→ Your next move: run `/ship-change`.** *"Now that the net's under you, this is the skill you'll use every single time you want to change anything — even a tiny tweak. It makes the change on a private copy first, lets you see it before anyone else does, and publishes only when you say so. Want to run it now?"*

**Tailor the *after-that* horizon to what `/readiness-check` found — read `readiness-check.md` for the required gate.** This is how the skills point onward based on how the app scored, not generically:
- **Required gate was 1** (small, no money, no one else's data): `/ship-change` *is* the destination. Tell them so — once it's a habit, they're fully set for where they want to be; nothing above is theirs to worry about.
- **Required gate was 2+** (real money / others' data, or pushing for growth — the danger-zone case): name the horizon honestly. `/ship-change` is still the immediate next move, but say the go-live gate is where their situation needs to get to, and all three of those skills (`/go-live-check`, `/qa-harness`, `/emergency-plan`) are ready now — so the small scope reads as "next, not last." **And set the expectation for what `/ship-change` does first:** because they're at Gate 2, its very first job is to build their *practice copy* (the **sandbox** from the picture above) — a private copy of the whole app that runs on their own computer — a one-time setup of about 15 minutes that it does for them right at the start, so it's ready for every change after and for the automatic checks in `/qa-harness`. Say so now so the up-front setup reads as planned, not as something going wrong: *"When you run `/ship-change`, the first thing it does is build your practice copy — about 15 minutes, one time, and it does all of it. After that every change is quick."*

If there's no `readiness-check.md` (front door was skipped), still offer `/ship-change`, and suggest running `/readiness-check` once to get the tailored horizon.

# Scope notes / known limits (mention if relevant; otherwise out of scope)

- **Mobile staging differs.** A phone app's "staging" is an over-the-air update channel or dev client, not a preview URL — handled in `/ship-change`, not here.
- **Sanitized data clones are deferred.** Realistic-but-safe staging data is a `/ship-change` concern; a half-correct sanitization script that leaks real PII is worse than an empty database, so it's opt-in there, not a default here.
- **Multi-deployment projects** (one backend, several front-ends) — this skill scopes to a single repo. Note the limitation if you see more than one deployment.
- **Scrubbing secrets from old git history** is a later polish; rotation (Step 2) is what actually closes the hole.
